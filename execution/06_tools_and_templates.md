## 1. 数据表格设计

### 总体架构

使用**飞书多维表格**作为核心数据管理工具，原因：
- 中文生态友好，权限管理灵活
- 支持关联字段，自动计算
- 与飞书文档/群组打通，协作成本低
- 免费版足够MVP使用（5人团队）

## 2. 工作流工具

### 2.1 工具选型

| 工具 | 用途 | 版本 | 月成本 |
|------|------|------|--------|
| **飞书多维表格** | 核心数据管理（5张表） | 免费版 | 0 |
| **飞书文档** | 方案文档、复盘报告 | 免费版 | 0 |
| **飞书群组** | 日常沟通 | 免费版 | 0 |
| **飞书项目** | 任务管理、进度看板 | 免费版 | 0 |
| **微信** | 外部沟通（创作者/商家） | — | 0 |
| **阿里云OSS** | 视频文件存储 | 标准版 | ~500元 |

**为什么用飞书而非Google Sheets/Notion**：
- 团队在国内，飞书访问稳定
- 飞书多维表格的公式字段、关联字段功能足够强大
- 与微信生态打通（飞书机器人可推送到微信）
- 免费版额度足够MVP使用

### 2.2 权限设置方案

**飞书空间结构**：
```
数字内容交易项目空间（根目录）
├── 数据库（5张多维表格）
│   ├── 创作者库 [运营读写，其他只读]
│   ├── 内容库 [运营读写，其他只读]
│   ├── 商家库 [运营读写，其他只读]
│   ├── 交易记录 [运营读写，财务只读]
│   └── 投放效果跟踪 [运营、数据分析师读写]
├── 文档（项目文档）
│   ├── 项目方案 [全员只读]
│   ├── 周复盘报告 [全员只读]
│   └── 创作者话术库 [运营读写]
└── 看板（飞书项目）
    └── 任务看板 [全员可见]
```

**外部人员访问**：
- 创作者/商家不直接访问飞书
- Phase 1（手动撮合）：运营通过微信沟通，手动录入飞书
- Phase 2（产品上线）：通过小程序/Web端访问，后台API连接飞书数据

### 2.3 日常更新流程

**创作者上架流程**（手动期 vs 产品期对比）

| 步骤 | Phase 1 手动操作 | Phase 2 产品化 |
|------|-----------------|---------------|
| 1. 创作者提交 | 微信群发抖音链接 | 小程序粘贴链接 |
| 2. 数据抓取 | 运营手动打开抖音复制数据 | 后台自动调用抖音API |
| 3. 视频下载 | 运营用工具下载无水印视频 | 后台自动下载并上传OSS |
| 4. 定价 | 运营根据经验人工定价 | 规则引擎自动定价 |
| 5. 录入系统 | 运营填写"内容库"表 | 自动写入数据库 |
| 6. 审核 | 运营看视频+改价格 | 运营在飞书表格中审核+调价 |
| 7. 上架 | 状态改为"已上架" | 状态自动改为"已上架" |

**商家购买流程**（手动期 vs 产品期对比）

| 步骤 | Phase 1 手动操作 | Phase 2 产品化 |
|------|-----------------|---------------|
| 1. 浏览素材 | 运营在Excel或飞书筛选，发给商家看 | 商家在Web端自己筛选 |
| 2. 预览视频 | 运营发带水印版微信视频 | Web端在线预览（带水印） |

| 6. 记录交易 | 运营手动填写"交易记录"表 | 自动写入数据库 |
| 7. 跟踪投放 | 运营Day1/3/7微信催数据 | 商家在Web端自行填写，或运营催促 |

## 7. 总结

### 关键决策

1. **工具选型**：飞书生态 + 微信沟通，避免工具分散
2. **权限分级**：内部全透明，外部（创作者/商家）只看自己
3. **自动化节奏**：Phase 1手动验证，Phase 2逐步自动化，不过度设计
4. **数据飞轮**：从Day 1开始积累"内容特征→跑量效果"数据，即使交易失败也能转型SaaS

### 核心指标优先级

**Gate 1决策（第4周末）看这3个指标**：
- 创作者上架率 ≥60%
- 商家购买意愿 ≥50%
- 实际成交笔数 ≥5笔

**Gate 2决策（第13周末）看这2个指标**：
- 跑量成功率 ≥ 自制素材的1.5倍（这是北极星）
- 商家复购率 ≥30%

### 工具成本

**前4周**：0元（飞书免费版+微信+人力）
**后3个月**：5000元/月（OSS存储+CLIP API+飞书付费版可选）

### 时间投入

- 创建所有表格：2小时
- 设置权限和看板：1小时
- 第一个自动化脚本：1天（工程师）
- 日常维护（运营）：2小时/天
